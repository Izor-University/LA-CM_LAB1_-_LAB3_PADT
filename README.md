# Полиморфные абстрактные типы данных (Лабораторная работа №3 — Matrixes)

## Сборка и запуск

Проект использует систему сборки CMake. Все команды выполняются из корневой директории проекта.

1. **Конфигурация и сборка проекта**
```bash
cmake -B build             # Конфигурация (выполняется единожды)
cmake --build build        # Компиляция проекта
```

2. **Запуск основной программы (Универсальный калькулятор)**
```bash
./build/app/LinearAlgebraApp # Или ./build/app/main в зависимости от ОС
```

3. **Запуск тестов и бенчмарков**
```bash
cd build && ctest --output-on-failure          # Запуск всех тестов
./build/tests/UnitTests                        # Запуск юнит-тестов (Google Test)
./build/tests/BenchmarkLR1                     # Запуск бенчмарков
```

## Архитектура и детали реализации

### Интеграция с предыдущими работами (Lab 2)
Для хранения элементов внутри базового класса `Matrix<T>` используется абстракция `MutableArraySequence<T>`. В консольном интерфейсе (`app/main.cpp`) глобальные реестры матриц и векторов построены на базе `LinkedList<MatrixRecord*>`. Это доказывает универсальность ранее написанных контейнеров и соблюдение принципа переиспользования кода (DRY).

### Принцип подстановки Лисков (LSP)
`Дочерний класс должен дополнять, а не замещать поведение родительского класса.`
Любая специализированная матрица может быть передана по указателю `IMatrix<double>*`. Полиморфизм гарантирует, что вызывая метод `Get(i, j)`, мы получим корректный элемент, независимо от того, хранится ли он в плотном массиве или вычисляется «на лету» (как нули в треугольных матрицах). Виртуальный деструктор интерфейса обеспечивает корректную очистку памяти при удалении объекта через базовый указатель.

### Тип Complex<T> и MathUtils
Класс комплексных чисел `Complex<T>` интегрирован в математическое ядро. Для поддержки алгоритмов, требующих сравнения величин (например, выбор главного элемента в Гауссе), использовано пространство имен `MathUtils::Abs()`. Оно унифицирует получение абсолютного значения (модуля) как для `double`, так и для `Complex`, что позволяет использовать `MatrixSolver` с обоими типами данных.

### IMatrix<T>
#### Схема наследования
```text
IMatrix<T>               ← Абстрактный базовый класс (интерфейс)
   ├── Matrix<T>         ← Прямоугольная плотная матрица (хранение в 1D)
   │   └── SquareMatrix<T>  ← Квадратная матрица (Trace, Identity, Det)
   ├── TriangularMatrix<T>  ← Оптимизировано под n*(n+1)/2 элементов
   └── DiagonalMatrix<T>    ← Хранение только n элементов диагонали
```

### Защита целостности памяти (Отсутствие operator[][])
В архитектуре принципиально не реализован доступ по двойным квадратным скобкам `matrix[i][j]` для записи. Возврат ссылки на внутреннюю память разрушил бы инкапсуляцию таких классов, как `DiagonalMatrix` и `TriangularMatrix`. Если пользователь попытается записать значение вне разрешенной зоны в обход проверки метода `Set()`, структура выбросит исключение. Для удобства чтения реализован безопасный `operator()(row, col)`, выполняющий математический пересчет 2D-индексов в 1D-индекс `row * cols + col`.

### MatrixSolver<T>
Статический класс-решатель, реализующий основные алгоритмы линейной алгебры. Все методы принимают объекты по значению или константной ссылке, активно используя move-семантику для исключения лишних копирований.

#### Реализованные алгоритмы решения:
1.  **SolveGaussNoPivot** (Наивный Гаусс)
    Классический алгоритм без перестановок. Выбрасывает исключение, если обнаруживает нулевой элемент на главной диагонали в процессе исключения, чтобы избежать скрытого деления на ноль.
    
2.  **SolveGaussPartialPivot** (Гаусс с выбором главного элемента)
    Более устойчивая версия, которая перед исключением переменной ищет максимальный по модулю элемент в текущем столбце и выполняет `SwapRows()`. Это минимизирует ошибку потери значащих цифр при работе с числами с плавающей запятой.

3.  **DecomposeLU и SolveUsingLU** (Разложение и решение)
    Факторизация матрицы $A$ в произведение нижней ($L$) и верхней ($U$) треугольных матриц. Разделение на этапы разложения и подстановки позволяет эффективно решать СЛАУ за $O(N^2)$ вместо $O(N^3)$, если меняется только вектор свободных членов $b$.

#### Фабрики и метрики
* **GenerateHilbertMatrix(n)**: Генерация плохо обусловленных матриц Гильберта для стресс-тестирования точности алгоритмов. Позволяет измерить относительную погрешность $\frac{||x_{approx} - x_{exact}||}{||x_{exact}||}$.
* **CalculateResidual**: Вычисление нормы невязки $||Ax - b||$ для оценки качества полученного решения.
